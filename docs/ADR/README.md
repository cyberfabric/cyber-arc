# Cyber Fabric ADR Set: Fabric Tool Architecture

This directory contains the ADR set for the Cyber Fabric tool itself — its operational model, core architecture, host integrations, implementation language, installation footprint, kit-based extensibility, and prompt generation. The ADRs are intentionally scoped to decisions about Fabric as a tool and platform, and deliberately do not impose architectural constraints on the skills that run on top of it. Each entry below states what was decided in the corresponding ADR, in paragraph form, without restating the options that were considered or the supporting evidence. Follow the linked ADR for the full context, drivers, alternatives, and consequences.

## Identifier Conventions

ADR artifact IDs use standard Cypilot-style identifiers, for example `cpt-cyber-fabric-adr-central-fabric-core`.

## Decisions

### [ADR-0001: Use a Unified Fabric Operational Model Across Host-Native Surfaces](0001-cpt-cyber-fabric-adr-unified-agent-first-tool-v1.md)

Cyber Fabric exposes one unified operational model across supported host-native surfaces rather than allowing every host to define its own meaning of planning, orchestration, prompt materialization, and setup. Supported hosts integrate Fabric through host-native plugins, extensions, or adapters, while a shared Fabric core preserves authoritative semantics for planning, orchestration, prompt materialization, validation, setup flows, and execution policy. The platform is agent-first rather than model-first: users select supported agent providers and execution backends through the same Fabric model rather than fragmented host-specific logic. A dedicated first-party TUI may exist as a thin utility for debugging, automation, or fallback access, but no such client is the architectural center, and supported hosts behave as views over one operational core rather than as independent runtimes.

### [ADR-0002: Centralize Orchestration in a Shared Fabric Core](0002-cpt-cyber-fabric-adr-central-fabric-core-v1.md)

Cyber Fabric centralizes its real operational behavior in one shared Fabric core that acts as the authoritative brain behind every supported host. The shared core owns pipeline planning and compatibility reasoning, orchestration state and execution lifecycle, host capability interpretation and backend selection policy, prompt materialization and layered prompt override evaluation, validation coordination and execution policy, and the inspectable history of why specific execution choices were made. Host adapters provide UX integration, local context capture, permission bridging, and host-specific rendering, and call into the shared core rather than reimplementing it. The core is not bound to one packaging form: it may be embedded as a library, run as a local daemon, be exposed through a service boundary, or combine those approaches as deployment context evolves, while keeping its semantic contract stable across local, embedded, and future server-backed forms.

### [ADR-0003: Integrate Cyber Fabric Through Host-Native Plugins and Adapters](0003-cpt-cyber-fabric-adr-host-native-plugins-and-adapters-v1.md)

Cyber Fabric is delivered into the environments where users already work — agent hosts such as Claude Code, Codex, OpenCode, and Pi, as well as IDE surfaces such as VS Code and future web or server clients — through host-native plugins, extensions, or adapters. Those adapters stay thin: they gather host session context, permissions, and local capabilities and present them to the shared Fabric runtime in a standard way; render Fabric actions and results in host-appropriate UX; and describe host features such as subagent support, background execution, tool calling, diff support, and review surfaces explicitly. Real orchestration semantics remain in shared Fabric behavior rather than inside per-host integrations, and new hosts can be added by implementing the adapter contract instead of cloning core logic. A dedicated first-party TUI is therefore not the primary user surface; thin operational clients may still exist for automation, diagnostics, or fallback access.

### [ADR-0004: Adopt TypeScript as the Primary Implementation Language for Cyber Fabric](0004-cpt-cyber-fabric-adr-typescript-primary-language-v1.md)

TypeScript is the primary implementation language for the main Cyber Fabric runtime. The shared Fabric core, orchestrator runtime logic, provider adapters, host integrations, thin operational utilities, web-facing surfaces, prompt-materialization tooling, and the main deterministic extension surface are all implemented in TypeScript so that one language and one set of typed contracts span UI, CLI, orchestration, and provider integrations. Other languages are allowed only as explicitly bounded external tools or specialized helpers when there is a clear capability or operational reason to step outside the primary runtime, and non-TypeScript helpers must remain explicitly bounded rather than redefining the system architecture.

### [ADR-0005: Install the System with a Minimal Project-Local Footprint](0005-cpt-cyber-fabric-adr-minimal-installation-footprint-v1.md)

Cyber Fabric is installed with a deliberately minimal project-local footprint. Only essential project-bound configuration and reviewable product artifacts live inside the repository; heavier runtime logic, generated agent wiring, caches, and optional integrations are externalized, shared, or materialized only when they are actually needed. A project can therefore adopt Cyber Fabric without introducing a large amount of generated or opaque local state, repository diffs stay focused on intentional product artifacts rather than runtime noise, and upgrades do not require broad repository rewrites for normal capability evolution. Local and centralized modes can share one conceptual model without enforcing the same on-disk footprint everywhere, while tooling clearly distinguishes required local state from external or generated state.

### [ADR-0006: Distribute Skills as Kit Packages with a Central Agent Registration Tool](0006-cpt-cyber-fabric-adr-kit-packaged-pluggable-skills-v1.md)

Skills are delivered as installable kit packages, and one central registration tool is the integration control plane that exposes those installed kits to agent runtimes. The central tool discovers installed kit packages, resolves their declared metadata, validates whether they can be exposed to a target agent runtime, and registers or generates the corresponding agent-facing skill entries, refreshing those registrations when kit sets change. New skill capabilities are therefore added primarily as kit packages rather than core product edits, and different agent runtimes are kept consistent because they are generated or synchronized from the same authoritative installed-kit set. Installation, update, and registration failures produce explicit validation errors instead of silent agent drift.

### [ADR-0007: Generate Agent Prompts Exclusively Through a Tool from Layered Markdown Prompt Files with Handlebars Block-Helper Instructions](0007-cpt-cyber-fabric-adr-tool-generated-layered-prompts-v1.md)

Agent prompts are not maintained as hand-edited monolithic text files. Each prompt is a markdown file with YAML frontmatter, and individual instructions inside the prompt are demarcated by Handlebars block helpers — for example `{{#instruction "<instruction-id>"}} ... {{/instruction}}` — where each block carries a stable identifier that survives layered overrides. A dedicated generation tool is the only path that materializes the agent-facing prompt. Override precedence is fixed and explicit: the project prompt layer takes priority over an outside-project user layer, which in turn overrides the default prompt layer shipped with the kit. Higher-priority layers may replace, append, remove, or reorder instructions by their identifiers using additional Handlebars block helpers, and the final prompt assembly preserves deterministic ordering after overrides are applied. Identity and conditional content share one Handlebars-based syntax — the scaffolding is visible in rendered markdown, giving reviewers an immediate visual cue that a block is dynamic.

### [ADR-0008: Deliver All Extensions and Prompts Through Kits with Global, Workspace, and Core-Bundled Scopes](0008-cpt-cyber-fabric-adr-kits-as-universal-extension-mechanism-v1.md)

All prompts and all extensions in Cyber Fabric are delivered through kits — there is no parallel extension mechanism, and a kit is THE entry point of the Fabric tool. A kit may live in one of three scopes: global (user-level, available across the user's projects), workspace / project (repo-local), or core-bundled (physically shipped with the Fabric tool itself as part of its release). Core-bundled kits do not have an independent semver: their version is the version of the Fabric tool that ships them, and they are upgraded together with Fabric itself. Standalone kits — both global and workspace-scoped — version themselves independently. The same packaging, validation, discovery, and registration tooling covers all three scopes so that the extension story remains a single coherent model rather than several parallel mechanisms.

### [ADR-0009: Define Kit Dependencies and Auto-Update Policy](0009-cpt-cyber-fabric-adr-kit-dependencies-and-auto-update-v1.md)

A kit may declare dependencies on other kits and may declare its own auto-update policy. Dependency expressions take exactly two semver forms: strict pinned (`==X.Y.Z`), which requires that exact version, and minimum / range (`>=X.Y.Z`), which accepts that version or any later compatible release. The auto-update policy takes the form `latest:[major=Y|N, minor=Y|N, patch=Y|N]`, where each of the three flags independently controls whether major, minor, or patch upgrades of that kit are pulled automatically. The flags are orthogonal so that a kit can, for example, opt into automatic patches while requiring manual review for minor and major changes. Both the dependency expressions and the update policy are read by the same central registration and installation tooling that already handles kit packaging.

### [ADR-0010: Adopt Handlebars for Prompt Templating with Layered Variable Resolution and Per-Instruction Extraction](0010-cpt-cyber-fabric-adr-prompt-templating-and-instruction-extraction-v1.md)

Prompts gain a templating layer powered by Handlebars — the same engine that already renders the `{{#instruction}}` block helpers from ADR-0007, so identity and conditional content share one runtime instead of running two parallel parsers. Conditional expressions live inside instruction blocks; the rest of the prompt remains plain markdown. Template variables are resolved by merging a global (user / installation) configuration file and a local (project / workspace) configuration file, with the local layer overriding the global layer per key. The CLI surface is symmetric with the script CLI of ADR-0017: `fabric prompt list [--kit <id>] [--json]` enumerates instructions across kits, `fabric prompt info <kit>:<id> [--json]` returns instruction metadata without rendering, and `fabric prompt get <kit>:<id> [args...]` walks the parsed Handlebars AST across the kit's prompt files, locates the matching instruction block helper, and returns only its rendered content. **Instruction ids are unique within the kit** so the `<kit>:<id>` scheme is unambiguous. `[args...]` carries `--var key=value` overrides; the full variable precedence is CLI overrides > local config > global config.

### [ADR-0011: Workspace as a First-Class Fabric Concept with init/add/remove/info Operations](0011-cpt-cyber-fabric-adr-workspace-as-first-class-concept-v1.md)

A Fabric workspace is a named set of repositories that share Fabric configuration, workspace-scoped kits (ADR-0008), and operational state. **The workspace is never a folder** — it is a logical grouping of repositories that may live in different folders on disk. The workspace is independent of any IDE — IDE-less CLI sessions, CI runners, and editor-specific environments all see the same workspace concept. The core CLI surface is `fabric workspace init [path]` to create one, `fabric workspace add <path>` and `fabric workspace remove <path>` to mutate the member list, `fabric workspace list [--json]` to enumerate every workspace known to the user across all projects, and `fabric workspace info [--json]` to emit structured workspace knowledge for the current workspace. **Workspace knowledge is global, not project-local** — it lives in a per-user global workspace registry (default `~/.fabric/workspaces/...`, exact format follow-on design) so a workspace is addressable by name regardless of CWD. Repositories may additionally contain a small local registration file at `.fabric/workspaces/<name>.toml` whose sole purpose is to make registering the workspace globally trivial — when Fabric encounters such a file in CWD or its ancestors and the workspace is not yet in the global registry, it auto-registers on first encounter. The local file is a convenience shortcut, not the source of truth, and stays small enough to satisfy ADR-0005's minimal-footprint discipline. `info` is positioned as the canonical knowledge primitive other tools call instead of parsing the registry directly. How each surface (CLI, REST API, Web UI, VS Code plugin) resolves *which* workspace a given operation targets is decided in ADR-0025.

### [ADR-0012: Bidirectional VS Code Workspace Interop with Init-Time Discovery and Multi-Config Handling](0012-cpt-cyber-fabric-adr-vscode-workspace-interop-v1.md)

Cyber Fabric integrates with VS Code workspaces (`.code-workspace`) bidirectionally. `fabric workspace import <path-to-.code-workspace>` generates a Fabric workspace from a VS Code workspace file, and `fabric workspace export --target=vscode [path]` generates a `.code-workspace` from the current Fabric workspace, so the two views are mutually derivable instead of accidentally divergent. `fabric workspace init` scans the target directory for `.code-workspace` files: zero found means an empty workspace is created; one found means Fabric offers to bootstrap from it; two or more found means Fabric presents an explicit choose-or-merge prompt where merging takes the union of folder lists and deduplicates overlapping paths. The interop is the first concrete instance of the host-adapter pattern from ADR-0003 applied at the workspace level; future ADRs may add JetBrains, Zed, or other IDE bridges following the same shape.

### [ADR-0013: Branches as a First-Class Concept](0013-cpt-cyber-fabric-adr-branches-as-first-class-concept-v1.md)

Branches are modeled as first-class structured data with their own CLI surface and workspace-aware filters. The CLI is `fabric branch list [--repo <r>] [--orphan] [--gone] [--stale[=<days>]] [--json]` for cross-repo aggregation across all workspace members, plus `info`, `create`, `delete`, `rename`, `track`, and `untrack`. Branches exist independently of worktrees — a branch can have no worktree and is still a first-class object. Operations explicitly out of scope are `merge`, `rebase`, `push`, `fetch`, and `cherry-pick`; those remain in `git` and `gh`. The structured `--json` output is the canonical machine contract that both the CLI's human rendering and the planned Fabric web UI consume from one model. Workspace-aware filters such as `--orphan` (no worktree), `--gone` (upstream remote branch deleted), and `--stale` (no new commits within a configurable window) answer real workspace-level questions that plain `git` cannot answer in one command.

### [ADR-0014: Worktrees as a First-Class Concept with Hybrid Ownership](0014-cpt-cyber-fabric-adr-worktrees-as-first-class-with-hybrid-ownership-v1.md)

Worktrees are modeled as first-class structured data with hybrid ownership: operations apply to every worktree regardless of who created it, while Fabric-created worktrees carry ownership metadata (creator id, session id, cleanup policy) that enables intelligent cleanup. The CLI is `fabric worktree list [--repo <r>] [--own | --external | --all] [--json]` for cross-repo aggregation, plus `info`, `create`, `drop`, `merge`, and `gc`. `merge` composes the branch model from ADR-0013 with the drop primitive — it merges a worktree's branch into a target branch and drops the worktree. `gc` is aggressive on Fabric-owned worktrees with policy metadata and conservative on external worktrees (those created by Claude Code's Agent tool, Ralph Loop, manual `git`, or `gh`), which always require explicit confirmation. `fabric workspace info` (ADR-0011) is extended to aggregate worktree state per workspace member repository through this same model. The implementation uses `simple-git` as the git library because no official git SDK exists for TypeScript and `isomorphic-git` and `nodegit` do not adequately support worktrees.

### [ADR-0015: Kit Configuration Storage with Local, Global, and In-Repo Scopes, Kit-Namespacing, and Backend-Agnostic SDK](0015-cpt-cyber-fabric-adr-kit-configuration-storage-v1.md)

Kits get a built-in key-value storage primitive shaped like browser localStorage: `get`, `set`, `delete`, `list`, JSON-serializable values, no schema management. Storage is automatically kit-namespaced so kits cannot collide. Three placement scopes are available per key: workspace-local (default `.fabric/storage/kits/<kit-id>/<collection>.json`), global (`~/.fabric/storage/kits/<kit-id>/<collection>.json`), and in-repo (`<repo-root>/.fabric/storage/kits/<kit-id>/<collection>.json`, deliberately committed). The kit author chooses scope per key; workspace-local is the default. The backend is intentionally hidden behind the SDK — files today, possibly a database tomorrow — so kit code does not change when the platform evolves. Operators get a uniform CLI surface (`fabric storage list / get / set / delete`) for inspection and management across kits.

### [ADR-0016: Kit Assets Addressable by Identifier](0016-cpt-cyber-fabric-adr-kit-assets-v1.md)

A kit can ship arbitrary files as assets — text, JSON, images, binary blobs, model files, fixtures — declared in the kit manifest with stable kit-author-chosen identifiers. `fabric assets get <kit> <id>` returns the on-disk path regardless of kit installation scope (core-bundled, global, workspace-scoped per ADR-0008); the SDK function `fabric.assets.get(kit, id)` returns the same path for kit code. `fabric assets list` enumerates assets across kits. Assets are read-only by contract; mutable kit-controlled state belongs in storage (ADR-0015). The two primitives are intentionally separated so that kit lifecycle, refactoring, and operator inspection each work cleanly.

### [ADR-0017: Scripts as Kit Resources](0017-cpt-cyber-fabric-adr-scripts-as-kit-resources-v1.md)

A kit can ship runnable code as first-class scripts declared in its manifest. `fabric script list [--kit <id>] [--json]` discovers scripts across installed kits; `fabric script info <kit>:<id>` returns a structured record (description, language, declared arguments); `fabric script run <kit>:<id> [-- <args...>]` executes the script with arguments forwarded. Scripts get SDK access to their own kit's storage (ADR-0015) and assets (ADR-0016) without having to know the kit id explicitly. The same surface is reachable from the Web UI (ADR-0018) per ADR-0001 and ADR-0002 — the Web UI calls Fabric core directly, not the CLI as a subprocess. Scripts are conceptually distinct from web extensions and dev tool plugins, which run in different runtimes.

### [ADR-0018: Fabric Web UI Built on frontx with Microfrontend-Based Kit Extensions](0018-cpt-cyber-fabric-adr-fabric-web-ui-on-frontx-v1.md)

Fabric ships a first-party Web UI launched by `fabric web run`, built on frontx (https://github.com/cyberfabric/frontx, develop / alpha channel). Frontx is chosen because of native microfrontend extensibility (kit web extensions plug in as microfrontends without a custom integration layer), bundled enterprise-grade features (auth flows, multi-tenancy hooks, audit and observability), cloud-readiness for a future centralized Fabric service, architectural alignment with cyber-fabric, and the fact that one framework powers both Fabric's own UI and kit web extensions instead of two. **The Web UI is a client of the Fabric REST API (ADR-0020)** — it does not call Fabric core directly and does not invoke the CLI as a subprocess; the REST API is itself a thin layer over Fabric core, so the same Web UI runs against a local REST API for desktop use or a remote REST API for cloud-deployed Fabric service without code changes. By default `fabric web run` connects to a local REST API bound to localhost on a free port chosen at runtime; explicit `--port <n>` overrides the local Web UI port. Kit web extensions reach Fabric capabilities and their kit's storage (ADR-0015) and assets (ADR-0016) through the same REST API.

### [ADR-0019: Dev Tool Plugins as Kit Resources with Delivery-Only Lifecycle](0019-cpt-cyber-fabric-adr-dev-tool-plugins-as-kit-resources-v1.md)

Kits can ship dev tool plugins for IDEs and agentic tools (VS Code extensions, JetBrains plugins, Cursor extensions, Claude Code plugins, agentic-tool plugins, etc.). Plugins are packaged in each host's native format (`.vsix`, `claude-plugin/`, JetBrains plugin, and so on) and Fabric is delivery-only: it installs and registers through the matching host adapter (ADR-0003), but plugin runtime (load, unload, update, restart, sandboxing, permissions) is owned by the host. Bumping a kit version re-registers its plugins through the host adapter; the host's own update path takes over from there. `fabric plugin list / info / install / uninstall` provides the operator CLI surface; the same operations are reachable from the Web UI (ADR-0018) per ADR-0001 and ADR-0002. Adding support for a new host means writing one host adapter, not redesigning the kit-resource concept.

### [ADR-0020: REST API as a Fabric Surface and Canonical Transport for Non-CLI Surfaces](0020-cpt-cyber-fabric-adr-rest-api-as-fabric-surface-v1.md)

Fabric exposes a REST API as a third sibling-surface to the CLI and the Web UI, AND as the canonical transport that all non-CLI surfaces (Web UI per ADR-0018, VS Code plugin per ADR-0021) use to reach Fabric core. By default the REST API binds to localhost on a free port chosen at runtime; explicit `--port <n>` overrides. The REST API is versioned via URL prefix (`/v1/...`) and serves an auto-generated OpenAPI specification at `/v1/openapi.json` so external tools can build typed clients. Workspace addressing per ADR-0025: workspace embedded in URL path as `/v1/workspaces/<name>/...` for workspace-scoped resources; general endpoints (`/v1/workspaces`, `/v1/health`, `/v1/openapi.json`) are workspace-agnostic. By default localhost requires no auth; remote enablement and authentication are explicit opt-in (follow-on design).

### [ADR-0021: VS Code Plugin as Fabric's First-Party Host Adapter for VS Code](0021-cpt-cyber-fabric-adr-vscode-plugin-fabric-host-adapter-v1.md)

Fabric ships an official `Fabric` VS Code extension that brings Fabric capabilities into VS Code through native VS Code patterns (Command Palette, Tree View, Side Panel, webview). The plugin is the concrete instantiation of the host-adapter pattern from ADR-0003 for VS Code — distinct from kit-shipped VS Code plugins (ADR-0019), which are user-extension code packaged in kits. The plugin is a thin client of the Fabric REST API (per ADR-0020); it does not call Fabric core directly, does not invoke the CLI as a subprocess, and works against a local Fabric daemon by default but can be configured to point at a remote REST API endpoint for cloud-deployed Fabric service. Per-window workspace context: one VS Code window ↔ one Fabric workspace, derived from the open folder or `.code-workspace` per ADR-0012. The plugin surfaces workspace info, branches, worktrees, commits, scripts (ADR-0017), and PRs (ADR-0024) through native VS Code UI; kit web extensions (ADR-0018) render as VS Code webview panels.

### [ADR-0022: Git Commits as a First-Class Concept](0022-cpt-cyber-fabric-adr-git-commits-as-first-class-concept-v1.md)

Commits are modeled as first-class structured data alongside branches (ADR-0013) and worktrees (ADR-0014). The data model includes sha, message, author (name and email), date, parents, files-changed summary, and an is-merge flag. The CLI is `fabric commit list [--repo <r>] [--branch <b>] [--since <t>] [--until <t>] [--author <a>] [--json]` for listing, `fabric commit info <sha> [--repo <r>] [--json]` for one commit, and `fabric commit log <range> [--repo <r>] [--json]` for ranges; cross-repo aggregation across the workspace is the default. Implementation uses `simple-git` (the same library chosen in ADR-0014, since worktrees and commits share one git library by design). Operations explicitly out of scope are rebase, cherry-pick, history rewriting, and signature verification — those remain in `git` directly. The same surface is reachable from the Web UI, REST API, and VS Code plugin per ADR-0001 and ADR-0002.

### [ADR-0023: Git Provider Abstraction with GitHub as the Default Provider](0023-cpt-cyber-fabric-adr-git-provider-abstraction-with-github-default-v1.md)

Fabric defines a **provider-agnostic interface** for git provider operations (pull-requests / merge-requests, reviews, comments, releases). The contract lives in Fabric core; concrete providers are kits per ADR-0008. **GitHub** is the default provider, shipped as a **core-bundled kit** per ADR-0008's core scope; its implementation uses `@octokit/rest` and `@octokit/graphql` — the **official GitHub TypeScript SDK** written and maintained by GitHub themselves. Other providers (GitLab, Bitbucket, Gerrit, Gitea, Forgejo, etc.) are workspace- or global-scoped kits implementing the same interface. A workspace can have multiple providers configured; provider selection per repo is determined by remote URL pattern matching (e.g. `github.com` → GitHub provider) or explicit configuration. Initial contract scope: pull-requests, reviews, comments, releases (minimum). Issues are out of scope of this ADR (extensible — providers can expose them as an optional capability). Webhooks, GitHub Apps, OAuth flows, and fine-grained permissions are out of scope (follow-on).

### [ADR-0024: Pull Requests as a First-Class Concept with Markdown-Aware Review](0024-cpt-cyber-fabric-adr-pull-requests-as-first-class-concept-v1.md)

Pull requests (and provider-specific equivalents — GitLab merge requests, Bitbucket PRs, etc.) are modeled as first-class structured data through the git provider abstraction (ADR-0023). The data model includes id, provider id, title, description (markdown), source/target branches, author, status, reviews, comments, file changes, and last commit. CLI surface: `fabric pr list/info/create/review/view/comment` covering listing, inspection, creation, approval/request-changes/comment, visualization, and line/file comments. **Markdown-aware review** is a required affordance, not optional: the Web UI provides side-by-side rendered preview of changed markdown files, line comments on rendered markdown (mapped back to source line), and suggestion blocks. This targets spec/ADR/documentation review as a primary use case, because Cyber Fabric is built around markdown artifacts. Same operations from CLI, Web UI, REST API, and VS Code plugin per ADR-0001 and ADR-0002; the Web UI is the richest visualization surface due to its rendering capabilities.

### [ADR-0025: Multi-Workspace Operation and Workspace Context Resolution Across Surfaces](0025-cpt-cyber-fabric-adr-multi-workspace-operation-and-context-resolution-v1.md)

Each Fabric surface (CLI, REST API, Web UI, VS Code plugin) operates against exactly one workspace per invocation or request, and resolves *which* workspace through context appropriate to that surface. **CLI**: explicit `--workspace <name>` flag wins; otherwise auto-detect by walking up from CWD until a `.fabric/workspaces/<name>.toml` registration file is found (Fabric registers the workspace in the global registry on first encounter if not already registered); otherwise command classification kicks in — general commands (`--help`, `--version`, `workspace list`, `workspace init`, etc.) run without a workspace, workspace-required commands fail with an explicit error pointing to `init`, `cd`, or `--workspace`. **REST API**: workspace embedded in URL path as `/v1/workspaces/<name>/<resource>`; general endpoints under `/v1/workspaces` (list/create), `/v1/health`, etc. are workspace-agnostic. **Web UI**: workspace selector in the UI; URL includes workspace id; switching reloads workspace-scoped views. **VS Code plugin**: per-window context — one VS Code window ↔ one Fabric workspace, typically derived from the open folder or `.code-workspace` per ADR-0012. Every Fabric command (CLI command and REST endpoint) declares whether it requires a workspace; default is workspace-required. Sticky-state and always-explicit alternatives were considered and rejected — the former hides state confusingly, the latter destroys CLI ergonomics for the common in-workspace case.

### [ADR-0026: Workspace-Aware Prompt Rendering with Auto-Injected Context and Init-Workspace Fallback](0026-cpt-cyber-fabric-adr-workspace-aware-prompt-rendering-v1.md)

`fabric prompt get` (and Web UI, REST API, VS Code plugin equivalents) is workspace-aware in two ways. **Auto-injection**: when a workspace is detected per ADR-0025, Fabric populates a `workspace` Handlebars variable from `fabric workspace info` so prompt templates can reference `{{workspace.name}}`, `{{#each workspace.members}}...{{/each}}`, etc., AND prepends an auto-generated workspace summary block to the rendered output. This guarantees that any agent receiving a rendered prompt sees workspace context up front (member repositories, branches, worktrees, scoped kits) without making a separate `fabric workspace info` call. **Init-workspace fallback**: when no workspace is detected and the requested prompt is workspace-required, Fabric returns a special init-workspace prompt — core-bundled per ADR-0008, itself workspace-agnostic — with deterministic guidance to `init` / `list` + `--workspace` / `cd` into a registered repo. Missing-workspace becomes a recoverable, agent-readable state instead of an empty response or raw error. Per-prompt opt-out of auto-injection is available through prompt frontmatter for the rare cases where workspace context is not wanted. Same behavior across all surfaces per ADR-0001 / ADR-0002.

### [ADR-0027: Secret Storage with OS-Keychain Backends and `fabric login` for Provider Authentication](0027-cpt-cyber-fabric-adr-secret-storage-and-fabric-login-v1.md)

Fabric provides a dedicated **secret storage** primitive distinct from ordinary key-value storage (ADR-0015), backed by OS-native keychains: macOS Keychain, Windows Credential Manager, Linux Secret Service (libsecret with GNOME Keyring or KWallet), with an encrypted-file fallback (AES-256-GCM) for headless and CI environments. The SDK is `fabric.secrets.get / set / delete / list(namespace, key)`; `list` returns key names only, never values. The CLI is `fabric secret list / get / set / delete` with masking by default and explicit `--reveal` after confirmation. Secret values are never returned over non-localhost REST API connections without TLS. Fabric also provides a top-level `fabric login` CLI verb for provider authentication: `fabric login git <provider-name>` invokes the git provider kit's auth flow (per ADR-0023) — OAuth device flow, browser-based OAuth, personal access token entry, etc., chosen by the provider kit — and stores the resulting credential in secret storage at a conventional namespace such as `git-providers/<provider-name>/token`. `fabric login` is extensible to non-git auth subjects (`fabric login claude`, `fabric login npm`, etc.) through the same dispatch model. Same login and secret-management operations available from Web UI (login buttons, masked list with reveal toggle), VS Code plugin (Command Palette and Fabric Side Panel), and REST API per ADR-0001 / ADR-0002.

### [ADR-0028: Per-Kit TypeScript Dependency Isolation via pnpm and Corepack with Process-Sandboxed Script Execution](0028-cpt-cyber-fabric-adr-per-kit-dependency-isolation-and-script-sandbox-v1.md)

Per-kit TypeScript dependencies are isolated using **pnpm via Corepack**: Fabric requires only Node as a hard prerequisite (Corepack is bundled with Node 16+), and Corepack auto-activates pnpm based on the `packageManager` field — no separate pnpm install step for users. The same pnpm tooling manages Fabric core's own development dependencies and kit-shipped dependencies. Each kit that declares dependencies in its manifest gets an isolated `node_modules/` via `pnpm install` scoped to the kit; pnpm's content-addressed global store deduplicates packages across kits, so different versions of the same library (kit A on `lodash@4.x` and kit B on `lodash@3.x`) coexist without disk waste, and pnpm's strict layout ensures a kit script's resolution sees only its declared deps. **Script execution is process-sandboxed**: `fabric script run <kit>:<id>` (per ADR-0017) spawns a child Node process with CWD = kit directory, `NODE_PATH` = kit's `node_modules`, and a whitelisted env-var set; a misbehaving script cannot bring down Fabric or interfere with sibling kits, and one kit's scripts cannot read another kit's secrets (ADR-0027) through inherited process environment. Stronger sandboxing (filesystem restrictions, network restrictions, CPU and memory limits, Node `--permission` flag, container-based isolation) is explicit follow-on design.

### [ADR-0029: Kit Marketplace Architecture: Git-Repo-Backed Kit Registries with Multi-Marketplace, Provider-Agnostic Distribution, and PR-Gated Versioned Publication](0029-cpt-cyber-fabric-adr-kit-marketplace-architecture-v1.md)

Cyber Fabric kit distribution uses a **git-repo-backed marketplace** model — no custom registry service. A marketplace is either a git repository or a local filesystem path holding a `manifest.yaml` at root; users register one or more locally at `~/.fabric/marketplaces/<name>/` (user-global) or `<repo>/.fabric/marketplaces/<name>/` (workspace-scoped). The official marketplace is core-bundled and tightly curated; arbitrary community, private, and local marketplaces can be added. Manifest entries are **versioned with pinned commit SHA + content SHA-256** per version: `mode: url` (kit lives in external repo) or `mode: local` (kit lives at `.fabric/kits/<id>/` in the marketplace repo); both modes share install logic via `git archive <commit>` and content-hash verification. **`fabric kit publish <git-url>`** creates a PR (or fork + PR for authors without write access) through ADR-0023's git provider abstraction, using credentials from ADR-0027; **`fabric kit publish --local <name>`** writes the entry directly with an explicit no-review-chain warning. Every install validates client-side that the downloaded content's SHA-256 matches the manifest — defense in depth even if marketplace CI or git provider is compromised. **Local-path marketplaces** (no git tracking) are protected by **TOFU manifest hashing** — Fabric records the manifest hash at `marketplace add` and prompts on change. Marketplace CI is the recommended convention for git-tracked marketplaces (auto-validates `sha256` on PR). Kit names enforced to `^[a-z][a-z0-9_-]*$`; symlink escape in local mode rejected. Multi-kit repos work naturally — one repo can be both a kit collection (`.fabric/kits/<name>/`) and a marketplace (`.fabric/marketplaces/<self-name>/manifest.yaml`). **Kit-to-kit dependencies** are declared per version in the manifest as `(marketplace, kit, version)` triples with exact version pinning; `marketplace` accepts `official` or a git URL (local-path marketplaces cannot be transitive dependency targets). `fabric kit install` resolves the dependency graph depth-first and installs deps before the requested kit, marking each install as `installed_as: direct` or `installed_as: dependency` in the local kit registry; conflicts (different versions of the same kit) abort with a report. `fabric kit uninstall` computes orphaned dependency installs (kits installed as deps with no remaining referrers) and prompts to remove them.

### [ADR-0030: Core-Bundled `core` Kit: Fabric Platform-Internal Prompts, Domain Model Skills, Guide Material, and Kit Development Toolkit](0030-cpt-cyber-fabric-adr-core-bundled-kit-core-v1.md)

Cyber Fabric ships **one core-bundled `core` kit** holding all Fabric-author-only platform content, organized by sub-namespace inside the kit. **`core:<name>`** — runtime prompts Fabric itself uses (`core:init-workspace` per ADR-0026 fallback, `core:workspace-summary` per ADR-0026 auto-injection, `core:login-device-flow` per ADR-0027, error templates). **`core:guide.<name>`** — educational material covering the Fabric domain model (`guide.fabric-overview`, `guide.workspaces`, `guide.kits`, `guide.marketplace`, `guide.prompts`, `guide.scripts`, `guide.git-domain`, `guide.secrets`). **`core:skill.<name>`** — agent-facing skills wrapping each domain entity (`skill.workspace`, `skill.kits`, `skill.marketplace`, `skill.branches`, `skill.worktrees`, `skill.commits`, `skill.prs`, `skill.scripts`, `skill.secrets`). **`core:kit.<name>`** — kit-development toolkit (`kit.scaffold`, `kit.author-prompt`, `kit.author-script`, `kit.author-web-extension`, `kit.author-dev-plugin`, `kit.document`, `kit.test`, `kit.publish-flow`). **`core:github.<name>`** — GitHub provider implementation per ADR-0023 (auth flows, provider operation scripts using `@octokit/rest` / `@octokit/graphql`, github-specific skills); a feature-area sub-namespace satisfying ADR-0023's commitment to ship GitHub with Fabric without a separate core kit. Octokit dependencies are declared in `core`'s `kit.yaml` and pinned in lock-step with Fabric releases.

### [ADR-0031: Prompt Type Taxonomy: Nine Types for Different Consumer Intents](0031-cpt-cyber-fabric-adr-prompt-type-taxonomy-v1.md)

Every prompt instruction (per ADR-0007) declares a `type` in its frontmatter, drawn from a **closed nine-type taxonomy**: **`template`** (content blueprint with placeholders), **`rules`** (guidelines / conventions / behavioral constraints), **`skill`** (reactive trigger registered with host agent tools per ADR-0030's `fab:<kit>:<name>` convention), **`agent`** (autonomous sub-agent registered with host agent tools), **`delegate`** (wraps invocation of external agent — Claude, Codex, Devin, Copilot CLI; `fabric prompt get` returns rendered content as the input; separate `fabric delegate run <kit>:<id>` actually invokes), **`checklist`** (structured verification / process steps), **`example`** (reference exemplar / canonical artifact), **`workflow`** (multi-phase markdown definition referencing other prompts; no dedicated Fabric execution engine — consumer iterates), and **`middleware`** (cross-cutting injection — selects targets by `target_type` and / or `target_id` glob, positions at `prepend` or `append`, composed automatically into target's rendering). One prompt = one type (mutex); the `type` field is authoritative; sub-namespaces (`core:skill.<name>`, etc.) are organizational convention. Middlewares do not match other middlewares (anti-recursion). ADR-0026's auto-injected workspace summary is conceptually a built-in middleware (`target_type: *`, `position: prepend`). Delegate target set is canonical (`claude`, `codex`, `devin`, `copilot-cli`) but extensible via kit-shipped adapters. Extensions to the type taxonomy require ADR amendments.

### [ADR-0032: Multi-Channel Fabric Distribution: npm, Homebrew, Scoop, VS Code Marketplace, and Agent-Tool Marketplaces](0032-cpt-cyber-fabric-adr-multi-channel-distribution-v1.md)

Fabric is distributed through **multiple channels**, each producing an equivalent install (same version, same `core` kit including the VS Code extension per ADR-0021 and GitHub provider per ADR-0023, same CLI / REST / Web UI surfaces). **MVP channels**: **npm / pnpm** (`pnpm add -g @fabric/cli`), **Homebrew** (`brew install fabric`; formula declares Node dep), **VS Code Marketplace** (extension install bootstraps Fabric core, including the `core` kit's VS Code extension per ADR-0030). **Follow-on channels**: Scoop (Windows), Claude Code / Codex marketplace (one-click add via agent tool, with skills registered per ADR-0030's `fab:<name>` convention), container images (Docker / OCI for CI), self-contained binary (`pkg` / `nexe` / Bun bundle for air-gapped scenarios). **Node 16+ is the universal hard prerequisite** per ADR-0004 / ADR-0028; each channel handles the prereq its own way (declare as dep, leverage existing, or ensure through installer). **Cross-channel duplicate-install detection** is warning-only; `fabric --version` reports the install channel. **Per-channel update** uses the native mechanism (`brew upgrade fabric`, `scoop update fabric`, `pnpm update -g @fabric/cli`, VS Code auto-update); a `fabric upgrade` dispatcher (which detects channel and forwards) is optional follow-on. All channels publish from the same source git tag; minor cross-channel publish lag is acceptable. **`core` ships no scripts** — deterministic kit-development operations (scaffolding, validation, hash computation, build, migration, diagnostics) are implemented as native `fabric` CLI commands (`fabric doctor`, `fabric kit scaffold`, `fabric kit validate`, `fabric kit build`, `fabric kit compute-hash`, `fabric kit add-prompt / add-script / add-web-extension / add-dev-plugin`, `fabric migrate`), not kit-shipped scripts. This is the defining property of `core` being the platform's own kit: `core` holds Fabric-author-only prompts and assets, while runtime determinism lives natively in Fabric. Kits ship `fabric script run <kit>:<id>` (ADR-0017) only for third-party operations Fabric itself cannot foresee. **`core` does ship one dev tool plugin** — the official **Fabric VS Code extension** (per ADR-0021) — packaged as a kit-shipped plugin per ADR-0019's host-native delivery, unifying the model: Fabric's own plugin uses the same ADR-0019 mechanism as third-party kit-shipped plugins. Future first-party dev tool plugins (JetBrains, Cursor, etc.) may be added to `core` following the same convention. MVP-blocking content is just `core:init-workspace` and `core:workspace-summary` (ADR-0026); everything else is natural growth. Opinionated workflow kits (planners, methodology toolkits) belong in marketplace, not in `core`. Skills and sub-agents from `core` (or any kit) register into host agent tools (Claude Code, Codex, etc.) under the convention **`fab:<kit>:<name>`**, with the `<kit>` segment **omitted for `core`** (so `core:skill.workspace` registers as `fab:skill.workspace`); the frontmatter field uses the quoted form `name: "fab:<kit>:<name>"`. The `fab:` prefix identifies content as Fabric-shipped, distinguishing it from host-native skills or other platforms' content.
